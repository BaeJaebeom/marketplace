# Save Session

## Overview

현재 세션의 작업 내용을 구조화된 MD 파일로 저장한다. 다음 세션에서 `/load-session`으로 자동 복원하거나 파일을 직접 제공하여 작업을 이어갈 수 있다.

## When to Use

- 세션을 종료하기 전에 작업 내용을 보존하고 싶을 때
- 긴 작업을 여러 세션에 걸쳐 이어서 해야 할 때
- `/save-session` 명령 실행 시

## 동작 절차

### 1. 정보 수집

다음 명령으로 변경 사항을 파악한다:

```bash
git status
git diff --stat
git branch --show-current
```

변경된 파일 경로에서 **서브프로젝트 단위**로 그룹핑한다 (예: `cms-api/`, `cms-fe/` 등).

### 2. 토픽명 자동 생성

세션에서 수행한 작업 내용을 기반으로 토픽명을 자동 생성한다. 사용자에게 묻지 않는다.
- git diff, 대화 맥락에서 핵심 키워드를 추출한다
- 영문 kebab-case로 간결하게 작성한다 (예: `fix-cart-api`, `add-product-filter`, `cms-worker-refactor`)
- 2~4단어 수준으로 작성한다

### 3. MD 파일 생성

`{프로젝트루트}/sessions/` 디렉토리에 아래 형식으로 저장한다.

**파일명**: `YYYY-MM-DD-HHmm-{토픽}.md`

**시간**: 현재 시각 기준 (년-월-일-시분)

### 출력 템플릿

```markdown
# Session Handoff: {토픽}

## 메타 정보
- 날짜: YYYY-MM-DD HH:mm
- 프로젝트: {프로젝트 루트 경로}
- 브랜치: {현재 브랜치}

## 변경된 프로젝트
- {서브프로젝트1}
- {서브프로젝트2}

## 작업 내용

### 1. {작업 제목}
**배경**: 왜 이 작업을 했는지 (요청 배경, 버그 원인 등)
**내용**: 구체적으로 무엇을 어떻게 변경했는지 상세하게 서술. 어떤 파일의 어떤 부분을 왜 그렇게 고쳤는지, 로직의 흐름이 어떻게 바뀌었는지 등을 포함한다.
**영향 범위**: 이 변경이 어떤 기능, 화면, API에 영향을 주는지

### 2. {작업 제목}
...

## 주요 결정사항
- {결정}: {선택한 이유}

## 미완성 작업 / 다음 단계
- [ ] {남은 할 일 1}
- [ ] {남은 할 일 2}

## 재개 시 참고사항
다음 세션에서 이 작업을 이어받을 때 반드시 알아야 할 컨텍스트.
예: 특정 파일을 먼저 읽어야 한다, 특정 서버를 띄워야 한다, 테스트가 아직 안 됐다 등.
```

### 4. 저장 확인

파일 저장 후 경로를 사용자에게 알려준다.

## 작업 내용 작성 기준

- **상세하게** 작성한다. 다음 세션의 Claude가 코드를 읽지 않고도 맥락을 이해할 수 있어야 한다.
- 각 작업마다 배경/내용/영향 범위 세 항목을 모두 채운다.
- 코드 변경의 핵심 로직을 설명한다 (단순 파일 목록이 아님).
- 결정사항에는 "왜 그 방식을 택했는지" 이유를 반드시 포함한다.

## 재개 방법 (다음 세션)

`/load-session` 명령으로 가장 최근 세션 파일을 자동으로 불러올 수 있다. 또는 사용자가 직접 저장된 MD 파일을 제공해도 된다.
